# 方法论：从需求推导概念模型

> 如何从业务需求推导出系统需要实现的模块/概念

## 一、核心区别

| 场景 | 方法 | 起点 | 终点 |
|------|------|------|------|
| 分析现有系统 | 归纳法 | 系统（结果） | 概念（原因） |
| 设计新系统 | 演绎法 | 需求（原因） | 系统（结果） |

**但核心思路相同**：
```
需求 → 核心流程 → 依赖关系 → 概念模型
```

---

## 二、推导公式

### 基本公式

```
需求 ──→ 核心循环 ──→ 依赖链 ──→ 概念模型
  │           │            │           │
  ▼           ▼            ▼           ▼
业务目标   数据流动      需要什么     实现的模块
```

### 详细推导步骤

```
Step 1: 明确需求
   这个系统要做什么？
   输入是什么？输出是什么？

Step 2: 抽象核心循环
   最基本的操作是什么？
   这个操作重复什么？

Step 3: 识别必要条件
   要完成核心循环，必须具备什么？
   （最少需要什么？）

Step 4: 补充辅助功能
   除了核心，还需要什么？
   （日志、错误处理、配置等）

Step 5: 组织层次结构
   这些概念如何分层？
   谁为谁服务？
```

## 三、实战推导：假设你要做一个"文本生成系统"

### Step 1: 明确需求

```
需求：接收用户文本输入，返回生成的文本

输入：文本 prompt（例如 "Hello"）
输出：生成的文本（例如 "World"）
```

### Step 2: 抽象核心循环

```
核心循环（自回归生成）：
while 未完成:
    1. 基于已有token，预测下一个token
    2. 将新token加入序列
    3. 判断是否结束

简化：
输入 → 预测 → 输出 → (回到输入)
```

### Step 3: 识别必要条件（关键！）

```
问：要"预测下一个token"，需要什么？

答：需要"模型"，模型知道如何从之前的token预测下一个

所以添加：Model（模型计算）

现在：
输入 → Model → 输出

---
问：Model 从哪来？

答：Model 需要"模型权重"，没有权重模型就是空的

所以添加：ModelWeights（模型参数）

现在：
输入 → Model(Weights) → 输出

---
问：Model 如何处理输入？

答：输入是文本，Model 需要"数值"，所以需要"编码"

所以添加：Tokenizer（分词） + Embedding（向量化）

现在：
输入 → Tokenize → Embedding → Model(Weights) → 输出

---
问：Model 输出是什么？

答：Model 输出是每个词的可能性（概率），需要"决策"选哪个

所以添加：Sampler（采样器）

现在：
输入 → Tokenize → Embedding → Model(Weights) → Sampler → 输出

---
问：输出是数字，如何变成文本？

答：需要"解码"

所以添加：Tokenizer（反向）/ 直接用 Sampler 的结果

现在：
输入 → Tokenize → Embedding → Model(Weights) → Sampler → Token → 文本输出
```

### Step 4: 补充辅助功能

```
核心流程有了，但系统不能只有"裸"功能，需要：

1. 资源管理
   - 模型计算需要 GPU → 需要 Device（设备抽象）
   - GPU 显存有限 → 需要 Allocator（显存分配）
   - 生成过程需要存储中间结果 → 需要 KV Cache

2. 错误处理
   - 可能出错 → 需要 Error Handling

3. 初始化
   - 系统启动需要准备 → 需要 Init / Logger

4. 配置管理
   - 需要知道用哪个模型、多大显存 → 需要 Config
```

### Step 5: 组织层次结构

```
按"谁为谁服务"分层：

┌─────────────────────────────────────┐
│ 服务层：如何接收请求、返回结果       │
│ Tokenizer, Sampler, API             │
├─────────────────────────────────────┤
│ 计算层：核心计算逻辑                 │
│ Model, Embedding, Transformer       │
├─────────────────────────────────────┤
│ 资源层：管理硬件资源                 │
│ Device, Allocator, KV Cache         │
├─────────────────────────────────────┤
│ 基础设施：底层支持                   │
│ Config, Logger, Error Handling      │
└─────────────────────────────────────┘
```

## 四、这个过程用到的分析方法

### 1. 逆推法（从目标倒推）

```
目标：输出文本
    ↓
需要：Sampler 把概率变成 token
    ↓
需要：Model 输出概率（Logits）
    ↓
需要：Model 有权重
    ↓
需要：Tokenizer 把文本变成数字
```

### 2. 必要条件法

```
问：没有这个，系统能工作吗？
- 没有 Tokenizer？→ 不能处理输入
- 没有 Model？→ 没有计算能力
- 没有 Sampler？→ 不知道选哪个
- 没有 Embedding？→ Model 不接受文本

这些就是"必要条件"
```

### 3. 边界分析法

```
每个组件的边界：
- Tokenizer：文本 ↔ 数字
- Embedding：数字 ↔ 向量
- Model：向量 ↔ 向量
- Sampler：向量 ↔ 数字

边界就是接口！
```

## 六、另一个例子：从需求推导"搜索系统"

### Step 1: 需求

```
需求：根据关键词找到相关文档
输入：搜索关键词
输出：相关文档列表
```

### Step 2: 核心循环

```
用户输入 → 匹配文档 → 排序结果 → 返回
```

### Step 3: 必要条件推导

```
问：如何"匹配文档"？
答：需要"索引"，知道哪些文档包含哪些关键词

所以添加：Index（索引）

问：如何"排序"？
答：需要"相关性计算"，算每个文档和查询的匹配程度

所以添加：Scorer（评分器）

问：文档从哪来？
答：需要"文档存储"

所以添加：Document Store
```

### Step 4: 补充

```
- 需要爬虫抓取文档 → Crawler
- 需要分词处理 → Tokenizer
- 需要缓存结果 → Cache
- 需要错误处理 → Error Handling
```

### Step 5: 分层

```
服务层：API, Request Handler
索引层：Index, Crawler
存储层：Document Store
基础层：Config, Logger
```

## 七、关键心法

### 心法一：先核心，后辅助

```
永远先问：没有这个，系统能跑起来吗？
- 能 → 辅助功能
- 不能 → 核心功能
```

### 心法二：逆推比顺推容易

```
顺推（从技术出发）：
我们有GPU → 可以做计算 → 做推理系统

逆推（从需求出发）：
要推理 → 需要模型 → 需要权重 → 需要加载

逆推更容易想清楚"为什么需要"
```

### 心法三：接口即边界

```
每个模块的输入输出就是接口

问：这个模块的输入是什么？输出是什么？
答不出来 → 还没想清楚这个模块
```

### 心法四：分层即解耦

```
想不清楚所有模块的关系？
先按"谁服务谁"分层
层内高内聚，层间低耦合
```

## 八、验证：你推导出的概念模型对吗？

### 验证问题

1. **完整性**：核心循环的每个步骤都有对应的模块吗？
2. **最小性**：删掉某个模块，核心循环还能工作吗？
3. **接口匹配**：前一个模块的输出是后一个模块的输入吗？
4. **层次清晰**：每一层都是为上一层服务吗？

### 实例验证

```
假设你推导出：
输入 → Tokenizer → Model → Sampler → 输出

问：Model 的输入是什么？
答：Token IDs

问：Tokenizer 的输出是什么？
答：Token IDs

→ 接口匹配 ✓

问：没有 Sampler 可以吗？
答：不行，不知道选哪个 token

→ Sampler 是核心 ✓
```

## 九、进阶：当你不知道"核心技术"是什么时

### 场景

你想做一个"AI 聊天机器人"，但不知道底层需要什么技术。

### 方法

```
1. 先明确"输入是什么，输出是什么"
   输入：文本，输出：文本
   
2. 问：输出是怎么生成的？
   答：用语言模型
   
3. 问：语言模型需要什么？
   答：需要模型文件、需要 GPU
   
4. 问：市面上的语言模型有哪些开源的？
   答：LLaMA, GPT, BLOOM...
   
5. 结论：
   核心技术是"语言模型推理"
   可以用 vLLM、Text Generation Inference 等框架
```

### 核心洞察

```
当你不知道核心技术是什么时：
1. 先明确输入输出
2. 研究"市面上的解决方案"
3. 把解决方案拆解成模块
```

## 十、总结

### 从需求到概念模型的流程

```
1. 明确需求
   ├── 输入是什么
   └── 输出是什么

2. 抽象核心循环
   ├── 最基本的操作
   └── 循环逻辑

3. 必要条件推导（逐层追问）
   ├── 要完成X，需要什么？
   ├── 没有Y能工作吗？
   └── 答案就是概念

4. 补充辅助功能
   ├── 资源管理
   ├── 错误处理
   └── 配置管理

5. 组织层次
   ├── 谁为谁服务
   └── 接口是什么
```

### 核心原则

```
记住这个公式：
需求 → 核心流程 → 必要条件 → 概念模型

逆推法：
目标 ← 需要什么 ← 怎么实现 ← 输入是什么
```

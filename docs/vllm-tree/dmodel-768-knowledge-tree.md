# d_model = 768 知识树

> 从"为什么是 768"出发的完整推导链

---

## 来时路

```
计算环节 → Transformer → 超参数设计 → d_model 选择
```

从 Transformer 的超参数设计出发，推导 d_model=768 的选择原因和影响。

---

## 根问题：为什么 d_model = 768？

### 第一问：d_model 是什么？

#### Q1.1: d_model 的定义
**A**: 
- d_model = 隐藏层维度 / 模型维度
- 也叫 embedding dimension 或 hidden size
- 表示每个 token 在模型内部被表示为多少维的向量

#### Q1.2: d_model 在哪里出现？
**A**: 一旦设定了 d_model = 768，以下地方的维度全部是 768：
- Embedding 输出
- 每一层 Attention 的输入输出
- 每一层 FFN 的输入输出
- 位置编码的维度
- LayerNorm 的参数（gamma 和 beta 向量）
- 残差连接的相加两端

**这就是为什么它是一个"全局统一的维度管道"**——整条管线都按这个宽度走数据。

#### Q1.3: 每个 token 真的是 768 维吗？
**A**: 
- **输入时**：一个 token 只是一个整数 ID（如 15）
- **经过 Embedding 后**：它变成一个 768 维的浮点数向量，比如 `[0.12, -0.45, 0.78, ...]`，共 768 个数
- **在 Transformer 内部流动时**：每个 token 始终被表示为一个 768 维向量（经过注意力、FFN、残差后依然是 768 维）
- **直到最后输出层**：才把这个 768 维向量投影到词表维度（10000 维），变成对每个词的预测分数

**所以，整个模型就是在不断变换这 768 个数字，让它们逐渐从一个"静态词义"变成"融合了上下文的动态语义"。**

---

### 第二问：为什么选择 768 这个具体数值？

#### Q2.1: 768 是推导出来的吗？
**A**: 不是。768 是人为设定的超参数，不是数学必然。

#### Q2.2: 768 叫什么？
**A**: 
- 在论文和代码里通常叫 d_model（隐藏层维度 / 模型维度）
- 有时也叫 hidden_size 或 embedding_dim

#### Q2.3: 为什么选 768 而不是其他数字？
**A**: 这是一个工程权衡，主要考虑以下因素：

| 因素 | 解释 |
|------|------|
| **2 的幂次 × 整数** | 768 = 12 × 64。64 是 d_head（每个注意力头的维度），12 是头数。硬件（GPU）对 2 的幂次运算更友好。 |
| **头数整除** | 必须能被注意力头数整除（768 / 12 = 64），否则多头切分时形状不匹配。 |
| **算力与性能平衡** | 更大的 d_model（如 1024、4096）→ 更强的表达能力，但计算量和显存也更大。768 在 BERT 时代（2018）是 GPU 显存（如 12GB）能跑得动且效果不错的平衡点。 |
| **历史遗产** | BERT 选了 768（12 头 × 64 维），成了事实标准。GPT 系列后来增大到 768、1600、4096、8192 等，但基本逻辑不变。 |

#### Q2.4: 768 的数学分解
**A**: 
```
768 = 12 × 64
     = num_heads × d_head
```
- 12 = 注意力头数
- 64 = 每个头的维度

这个分解确保了多头注意力可以正确切分和合并。

### 第三问：不同模型的 d_model 选择规律是什么？

#### Q3.1: 经典模型的 d_model 值
**A**: 

| 模型 | d_model | 头数 | d_head | 备注 |
|------|---------|------|--------|------|
| BERT-base | 768 | 12 | 64 | 经典配置 |
| BERT-large | 1024 | 16 | 64 | 更大模型 |
| GPT-2 small | 768 | 12 | 64 | 与 BERT-base 相同 |
| GPT-3 (175B) | 12288 | 96 | 128 | 超大模型 |
| LLaMA-7B | 4096 | 32 | 128 | 现代大模型 |

#### Q3.2: 选择规律
**A**: 
```
d_model = num_heads × d_head
```
- 每头维度通常是 64 或 128
- 头数通常是 8、12、16、32、96 等
- d_model 必须能被头数整除

#### Q3.3: 为什么 d_head 通常是 64 或 128？
**A**: 
- **64**: 经典选择，在 BERT/GPT-2 时代广泛使用
- **128**: 现代大模型（如 LLaMA）的选择，提供更强的表达能力
- 这两个值都是 2 的幂次，硬件友好

### 第四问：如果改成 256 或 4096 会怎样？

#### Q4.1: 改成 256 会怎样？
**A**: 
- **表达能力下降**：256 维的向量能表示的信息量有限，模型可能难以学习复杂的语言规律
- **计算量减少**：Attention 的计算复杂度是 O(T²·d)，参数量是 O(d²)，256 的参数量是 768 的约 1/9
- **适用场景**：小模型、资源受限环境、快速原型验证

#### Q4.2: 改成 4096 会怎样？
**A**: 
- **表达能力增强**：4096 维能表示更丰富的语义信息
- **计算量剧增**：参数量是 768 的约 28 倍，显存需求也大幅增加
- **适用场景**：大模型、需要更强表达能力的任务、有充足 GPU 资源

#### Q4.3: 如何选择合适的 d_model？
**A**: 
```
选择 d_model 时需要考虑：
1. GPU 显存限制
2. 训练速度要求
3. 任务复杂度
4. 模型规模目标
```

**经验法则**：
- 小模型（<1B 参数）：256-768
- 中等模型（1B-10B）：768-2048
- 大模型（>10B）：2048-8192

---

### 第五问：d_model 如何影响整个模型架构？

#### Q5.1: d_model 对参数量的影响
**A**: 
```
参数量 ≈ 12 × d_model² + 2 × vocab_size × d_model
```
- d_model 的平方项占主导
- 768 → 4096，参数量增加约 28 倍

#### Q5.2: d_model 对计算量的影响
**A**: 
```
Attention 计算量：O(T² × d_model)
FFN 计算量：O(T × d_model × d_ff)
```
- d_model 越大，计算量越大
- 但计算量增长是线性的，参数量增长是平方的

#### Q5.3: d_model 对显存的影响
**A**: 
```
显存占用主要来自：
1. 模型参数：O(d_model²)
2. 激活值：O(B × T × d_model)
3. KV Cache：O(B × T × d_model)
```
- d_model 越大，显存需求越大
- KV Cache 的显存占用与 d_model 成正比

---

### 第六问：d_model 与其他超参数的关系？

#### Q6.1: d_model 与 num_heads 的关系
**A**: 
```
d_model = num_heads × d_head
```
- 必须满足整除关系
- 通常 num_heads 是 8、12、16、32 等
- d_head 通常是 64 或 128

#### Q6.2: d_model 与 d_ff 的关系
**A**: 
```
d_ff = 4 × d_model
```
- FFN 的中间层维度通常是 d_model 的 4 倍
- 例如：d_model=768 → d_ff=3072

#### Q6.3: d_model 与 vocab_size 的关系
**A**: 
```
Embedding 参数量：vocab_size × d_model
LM Head 参数量：vocab_size × d_model
```
- 通常与 Embedding 共享权重
- vocab_size 通常远大于 d_model

---

### 第七问：如何验证 d_model 的选择是否合理？

#### Q7.1: 训练时的指标
**A**: 
- **Loss 下降速度**：d_model 太小可能收敛慢或陷入局部最优
- **最终性能**：d_model 太大可能过拟合，太小可能欠拟合
- **训练速度**：d_model 越大，训练越慢

#### Q7.2: 推理时的指标
**A**: 
- **显存占用**：d_model 越大，推理时显存需求越大
- **延迟**：d_model 越大，计算延迟越高
- **吞吐量**：d_model 越大，batch size 可能需要减小

#### Q7.3: 实验验证方法
**A**: 
```
验证步骤：
1. 固定其他超参数，只改变 d_model
2. 在验证集上评估性能
3. 选择性能-成本平衡点
```

## 终极总结

### d_model = 768 的本质

**768 只是一个经典的历史选择，不是神谕。**

它满足以下条件：
1. 能被注意力头数整除（768 / 12 = 64）
2. 是 2 的幂次 × 整数（12 × 64）
3. 在 BERT 时代是 GPU 显存和性能的平衡点
4. 成为了事实标准，被广泛采用

### 选择 d_model 的关键原则

| 原则 | 说明 |
|------|------|
| **整除性** | d_model % num_heads == 0 |
| **硬件友好** | 优先选择 2 的幂次组合 |
| **资源平衡** | 在显存、速度、性能之间权衡 |
| **任务适配** | 根据任务复杂度调整 |

### 不同规模的推荐配置

| 模型规模 | d_model | num_heads | d_head | d_ff |
|----------|---------|------------|--------|------|
| 小型 | 256-512 | 8-12 | 32-64 | 1024-2048 |
| 中型 | 768-1024 | 12-16 | 64 | 3072-4096 |
| 大型 | 2048-4096 | 16-32 | 128 | 8192-16384 |
| 超大 | 8192-12288 | 32-96 | 128 | 32768-49152 |

### 关键洞察

**d_model 是模型"信息管道"的宽度**：
- 太窄（如 256）：信息流受限，表达能力不足
- 太宽（如 8192）：信息流冗余，计算和存储成本高
- 合适（如 768）：在资源约束下最大化表达能力

**整个模型就是在不断变换这 d_model 个数字**：
- 从静态词义（Embedding）
- 到上下文语义（Attention + FFN）
- 到预测概率（LM Head）

**768 不是数学必然，而是工程选择。** 你完全可以改成别的数，只要满足 `d_model % num_heads == 0`，并在资源、性能、效果之间找到平衡。

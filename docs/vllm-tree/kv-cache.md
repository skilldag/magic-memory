### 问题1：为什么需要缓存？

**问题**：为什么不能每次都重新计算？

**原因分析**：
```
1. Attention 需要计算 QKᵀ
2. 每个位置都要计算之前的所有位置
3. 重复计算太慢
4. 需要缓存 K 和 V
```

**解决**：KV Cache

### 问题2：如何管理？

**问题**：如何管理缓存的分配和释放？

**原因分析**：
```
1. 缓存需要分配显存
2. 请求完成后需要释放
3. 多请求需要共享
4. 需要管理生命周期
```

**解决**：KVCacheManager

## 第二层推导：KV Cache 结构

### 2.1 KV Cache 详细推导

```
[KV Cache]
    │
    ├── 问题：存储什么？
    │       解决：K 和 V 矩阵
    │
    ├── 问题：如何组织？
    │       解决：按位置存储
    │
    └── 问题：如何访问？
            解决：按位置索引
```

**为什么叫 KV Cache？**
- Key-Value Cache
- 缓存 Attention 的 K 和 V

### 2.2 KVCacheManager 详细推导

```
[KVCacheManager]
    │
    ├── 问题：如何分配？
    │       解决：按需分配
    │
    ├── 问题：如何释放？
    │       解决：请求完成释放
    │
    └── 问题：如何共享？
            解决：引用计数
```

**为什么需要 Manager？**
- KV Cache 只是数据结构
- Manager 负责管理生命周期

## 第三层推导：实现细节

### 3.1 KV Cache 存储

**问题**：KV Cache 如何存储？

**解决**：
```
1. 按位置存储
2. 每个位置存储 K 和 V
3. 形状：[num_tokens, num_heads, head_dim]
```

### 3.2 引用计数

**问题**：如何管理共享？

**解决**：
```
1. 每个位置有引用计数
2. 增加引用：++
3. 减少引用：--
4. 计数为0时释放
```

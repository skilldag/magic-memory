# Scheduler 深层推导

> 从"多请求如何调度"往下推，推到不能再分

---

## 表层问题

**输入**：多个请求 [req1, req2, ..., reqN]
**输出**：执行顺序
**问题**：请求先后的顺序如何决定？

---

## 第一层推导

```
    [多请求 → 执行]
          │
    ┌─────┴─────┐
    │           │
[优先级] [资源分配]
    │           │
    ▼           ▼
[FCFS/Priority] [GPU分配]
    │           │
    └─────┬─────┘
          │
    [Scheduler]
```

| 层级 | 问题 | 解决 | 概念 |
|------|------|------|------|
| 表层 | 请求先后的顺序？ | 调度 | Scheduler |
| 底层 | 调度策略 | FCFS/优先级 | Policy |
| 底层 | GPU不够 | 抢占 | Preemption |
| 底层 | 如何选择 | 亲和度 | Affinity |

---

## 第二层推导：调度策略

### 2.1 FCFS (First-Come-First-Serve)

```
[FCFS]
    │
    ├── 先来先服务
    │       │
    │       优点：简单、公平
    │       缺点：可能卡住
    │
    └── 问题：大请求等太久
            解决：超时中断
```

### 2.2 Priority

```
[Priority]
    │
    ├── 优先级队列
    │       │
    │       问题：低优先级饿死？
    │               解决：时间片轮转
    │
    └── 优先级来源：
            ├── 显式指定
            └── 隐式推断
```

### 2.3 SJF (Shortest Job First)

```
[SJF]
    │
    ├── 优先处理短请求
    │       │
    │       优点：周转快
    │       缺点：长请求等太久
    │
    └── 问题：如何预估长度？
            解决：prompt长度估计
```

---

## 第三层推导：Preemption & Affinity

### 3.1 Preemption (抢占)

```
[Preemption]
    │
    ├── 问题：GPU不够？
    │       解决：抢占
    │
    ├── 类型：
    │       ├── 回收
    │       └── 交换
    │
    └── 问题：恢复？
            保存KV cache
```

### 3.2 Affinity (亲和)

```
[Affinity]
    │
    ├── 问题：如何选请求填GPU？
    │       解决：亲和度
    │
    ├── 亲和因素：
    │       ├── 模型相同
    │       ├── 前缀相同
    │       └── 序列长度接近
    │
    └── 目标：最大化缓存命中
```

## 完整推导树

```
Scheduler
    │
    ├── [Policy]
    │       ├── FCFS
    │       ├── Priority
    │       └── SJF
    │
    ├── [Preemption]
    │       ├── 回收
    │       ├── 交换
    │       └── 恢复
    │
    └── [Affinity]
            ├── 模型亲和
            ├── 前缀亲和
            └── 长度亲和
```

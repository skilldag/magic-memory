# 问题发现方法论

> 如何从表象问题追溯到本质问题？如何发现需要解决的关键问题？

---

## 核心问题：如何发现问题？

**不是**有了概念再想问题，**而是**先有问题再有概念。

---

## 四大问题发现方法

### 方法1：流程分析法

**核心**：从工作流程中自然发现问题

**步骤**：
```
1. 找到系统的"根循环"
2. 列出每个环节
3. 问：这里的输入是什么？输出是什么？
4. 问：这个转换可能有什么问题？
```

**示例**：

```
根循环：输入 → 编码 → 计算 → 采样 → 判断 → 输出
              │        │        │
              ▼        ▼        ▼
         输入处理？ 计算复杂？ 采样策略？
              │        │        │
              ▼        ▼        ▼
         Tokenizer  Attention Sampler
```

### 方法2：瓶颈分析法

**核心**：从性能瓶颈倒推问题

**步骤**：
```
1. 测量/观察：哪里慢？哪里占内存？
2. 问：为什么慢？为什么占内存？
3. 层层追问：直到不能再问
4. 找到根本原因
```

**示例**：

```
现象：生成速度慢
    │
    哪里慢？自回归一次生成一个token
    │       │
    为什么慢？每个token都要完整计算
    │           │
    哪里最慢？Attention计算
    │           │
    为什么慢？O(N²)复杂度
    │           │
    为什么是N²？需要存储QKᵀ矩阵
    │           │
    显存爆炸？N越大，矩阵越大
```

### 方法3：对比分析法

**核心**：从"现状 vs 理想"找差距

**步骤**：
```
1. 确定"理想状态"是什么
2. 分析当前与理想的差距
3. 差距就是问题
4. 问题→概念
```

**示例**：

```
理想：高吞吐量 + 低延迟 + 低显存
        │        │        │
        ▼        ▼        ▼
    当前：低吞吐 当前：高延迟 当前：高显存
        │        │        │
        ▼        ▼        ▼
    差距：批处理 差距：串行   差距：碎片化
        │        │        │
        ▼        ▼        ▼
    Scheduler  Prefill/   PagedAttention
              Decode分离
```

### 方法4：需求驱动法

**核心**：从用户需求追溯问题

**步骤**：
```
1. 用户想要什么？（输入需求）
2. 系统需要提供什么？（输出需求）
3. 中间可能有什么问题？
4. 问题→概念
```

**示例**：

```
用户需求：支持更长上下文（输入）
    │
    需要处理更长序列
    │       │
    序列长了有什么问题？显存爆炸
    │           │
    解决？分页管理
            │
            PagedAttention
```

## 问题分类框架

### 1. 功能性问题

**定义**：功能上的缺失或错误

```
表现：功能不可用/不正确
发现：测试失败/用户反馈
解决：添加/修复功能
```

### 2. 性能问题

**定义**：速度/资源效率问题

```
表现：慢/占用高
发现：benchmark/监控
解决：优化算法/缓存
```

### 3. 扩展性问题

**定义**：scale相关问题

```
表现：大数据/高并发失效
发现：压力测试
解决：分布式/分片
```

### 4. 可用性问题

**定义**：易用性相关

```
表现：难用/不稳定
发现：用户反馈
解决：API优化/降级
```

## 发现问题的检查清单

当你要分析一个新系统时，检查以下问题：

### 基础层检查

```
□ 输入是什么？输出是什么？
□ 核心循环是什么？
□ 每个环节可能的瓶颈在哪里？
```

### 性能层检查

```
□ 什么操作最慢？
□ 什么资源最占？
□ 复杂度是多少？(O(N)? O(N²)? O(N³)?
```

### 扩展层检查

```
□ 大数据会怎么样？
□ 高并发会怎么样？
□ 分布式会怎么样？
```

### 用户层检查

```
□ 用户最关心什么？
□ 用户会遇到什么问题？
□ 竞品/理想方案的差距在哪？
```

## 记住方法

```
问题发现 = 流程分析 + 瓶颈分析 + 对比分析 + 需求驱动
         │           │          │          │
         └───────────┴──────────┴──────────┘
                          │
                    多个维度交叉验证
```

**核心原则**：不要只从一个角度发现问题，要多维度交叉验证。

---

## 检查流程

```
1. 先流程分析：理解系统怎么工作
2. 再瓶颈分析：找到哪里慢/占资源
3. 做对比分析：理想vs现状差距
4. 最后需求分析：用户要什么
5. 综合：问题→概念映射
```

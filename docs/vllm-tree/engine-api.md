## 第一层推导：为什么需要 Engine API？

```
[用户请求 → API]
      │
  ┌───┴───┐
  │       │
[问题1] [问题2] [问题3]
  │       │       │
  ▼       ▼       ▼
[如何定义] [如何调用] [如何返回]
  │       │       │
  └───┬───┴───┘
      │
  [Engine API]
```

### 问题1：如何定义？

**问题**：如何定义 API 接口？

**原因分析**：
```
1. 用户需要调用模型
2. 需要定义接口
3. 需要统一规范
4. 需要易于使用
```

**解决**：Engine API

### 问题2：如何调用？

**问题**：如何调用引擎？

**原因分析**：
```
1. 用户通过 API 调用
2. 需要参数验证
3. 需要错误处理
4. 需要异步处理
```

**解决**：API 层

### 问题3：如何返回？

**问题**：如何返回结果？

**原因分析**：
```
1. 生成需要时间
2. 需要流式返回
3. 需要错误处理
4. 需要状态跟踪
```

**解决**：流式返回

## 第二层推导：API 结构

### 2.1 Engine API 详细推导

```
[Engine API]
    │
    ├── 问题：如何定义接口？
    │       解决：RESTful / gRPC
    │
    ├── 问题：如何验证参数？
    │       解决：参数校验
    │
    └── 问题：如何处理错误？
            解决：错误处理
```

**为什么叫 Engine API？**
- 引擎（engine）的 API
- 对外提供的服务接口

### 2.2 流式返回详细推导

```
[流式返回]
    │
    ├── 问题：如何流式返回？
    │       解决：WebSocket / SSE
    │
    ├── 问题：如何分块返回？
    │       解决：逐 token 返回
    │
    └── 问题：如何处理中断？
            解决：异常处理
```

## 第三层推导：实现细节

### 3.1 参数验证

**问题**：如何验证参数？

**解决**：
```
1. 检查参数类型
2. 检查参数范围
3. 提供错误信息
4. 返回 400 错误
```

### 3.2 错误处理

**问题**：如何处理错误？

**解决**：
```
1. 捕获异常
2. 记录日志
3. 返回错误信息
4. 清理资源
```

## 记住

```
Engine API = 接口定义 + 参数验证 + 流式返回
            │
            └── 核心问题：如何对外提供易用、可靠的 API 服务？

为什么需要 API？

为什么需要流式返回？
```

### 问题1：为什么需要词嵌入？

**问题**：为什么不能直接用数字ID？

**原因分析**：
```
1. Token ID 是离散的（0, 1, 2, ...）
2. 离散数字没有语义关系
3. 模型需要连续的向量表示
4. 需要学习词的语义
```

**对比**：

| 表示方式 | 优点 | 缺点 | 问题 |
|----------|------|------|------|
| **One-hot** | 简单 | 维度太高（词表大小） | 计算量大 |
| **ID直接** | 简单 | 没有语义 | 无法学习 |
| **Embedding** | 语义丰富 | 需要训练 | **需要学习** |

**结论**：需要 Embedding 来学习词的语义表示

### 问题2：为什么需要位置编码？

**问题**：为什么不能只编码词本身？

**原因分析**：
```
1. Transformer 是并行处理所有token
2. 没有位置信息
3. "I love you" 和 "you love I" 一样
4. 需要区分位置
```

**解决**：位置编码（Positional Encoding）

## 第二层推导：词嵌入

### 2.1 绝对位置编码

```
[绝对位置]
    │
    ├── Sinusoidal

    │               解决：不同频率的正弦
    │
    └── 可学习
            └── 问题：位置需要学吗？
```

### 2.2 相对位置编码

```
[相对位置]
    │
    ├── 问题：位置相对关系重要？
    │       解决：编码相对偏移
    │
    └── 问题：长序列泛化？
            解决：截断到最大距离
```

### 2.3 RoPE (旋转位置编码)

```
[RoPE]
    │
    ├── 问题：如何融合位置和信息？
    │       解决：旋转矩阵
    │
    └── 问题：计算效率？
            解决：逐token应用
```

## 第三层推导：实现细节

### 3.1 词嵌入矩阵

```
shape: [vocab_size, hidden_dim]

存储:
    │
    ├── 问题：太大怎么办？
    │       解决：投影
    │
    └── 问题：并行查表？
            解决：gather操作
```

### 3.2 位置编码计算

```
正弦编码公式:
pos_emb[pos, 2i] = sin(pos / 10000^(2i/d))
pos_emb[pos, 2i+1] = cos(pos / 10000^(2i/d))
```

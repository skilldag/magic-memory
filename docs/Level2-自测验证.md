## 答案提示（先思考再查看）

### 10-13：模型加载与运行

**答案 1**：ModelRegistry 提供模型名到模型类的映射，让系统知道用什么类来加载和运行
**答案 2**：ModelLoader 负责加载权重，ModelRunner 负责执行推理，Loader 是 Runner 的前置步骤
**答案 3**：Model 类是计算核心本身，负责前向传播计算

### 14-16：计算核心

**答案 1**：在 GPU 上计算，因为后续计算都在 GPU 上，避免 CPU-GPU 数据传输开销
**答案 2**：Transformer 层是无状态的（权重固定），KV Cache 在每层产生并存储
**答案 3**：传统 Attention 需要为每个 Request 预分配固定长度的 KV 显存，导致碎片化和浪费

### 16-19：PagedAttention 核心机制

**答案 1**：把 KV Cache 分成固定大小的块，按需分配，像操作系统的虚拟内存
**答案 2**：因为按需分配，不需要预分配固定长度，避免了内部碎片和外部碎片
**答案 3**：映射虚拟块到物理块，类似但比操作系统页表简单（不需要换页）
**答案 4**：默认固定（16 tokens/块），可变会导致管理复杂度增加
**答案 5**：维护空闲块链表和已分配块的引用计数
**答案 6**：触发缓存驱逐，释放旧请求的块，或者等待（看调度策略）

### 20-24：采样与生成

**答案 1**：通常在 CPU 上（采样逻辑简单），但也可以在 GPU 上（使用 CUDA）
**答案 2**：temperature=0 相当于 greedy 采样，top_k 和 top_p 在此时无效
**答案 3**：由模型输出层产生，维度等于 vocab_size（词表大小）
**答案 4**：自回归生成需要前一个 token 的输出作为下一个的输入，依赖关系导致串行

### 25-29：执行优化

**答案 1**：Forward Pass 是一次前向计算，Decode Step 是生成一个 token 的完整过程（包含 Forward Pass）
**答案 2**：Prefill 阶段处理所有输入 token（并行），Decode 阶段每次只处理 1 个新 token（串行）
**答案 3**：通过分块计算和融合核函数，减少 HBM 访问，算法复杂度更低
**答案 4**：会变，误差来自量化精度损失（从 FP16 到 INT8）
**答案 5**：对于超大模型，需要分片加载或使用模型并行

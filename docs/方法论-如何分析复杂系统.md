# 方法论：如何分析复杂系统

> 从 0 推演出完整知识体系的操作指南

## 核心理念

分析一个复杂系统（比如 vLLM）不是靠"硬记"，而是靠**推导**。当你理解了系统的底层逻辑，50 个概念会自然浮现，不需要一个个去背。

这个方法论可以应用于任何复杂系统：操作系统、数据库、深度学习框架、网络协议栈等。

---

## 一、五大分析方法

### 方法一：依赖分析（Dependency Analysis）

**核心问题**：要实现 X，需要先实现什么？

**操作步骤**：

1. **找到最底层的"不能依赖任何东西"的概念**
   - 问：这个东西需要其他东西才能存在吗？
   - 如果不需要，那就是最底层

2. **逐层向上找依赖**
   - 对于每个概念，问：它的输入是什么？这些输入从哪来？
   - 那些"提供输入"的就是它的依赖

3. **画出依赖图**
   - 节点：概念
   - 边：依赖关系（X 需要 Y 才能工作）

**vLLM 例子**：

```
问：PagedAttention (16) 依赖什么？
答：它需要 KV Cache 存储数据，需要 Block Table 映射地址，需要 GpuAllocator 分配显存
    ↓
依赖：9(KV Cache) → 17(Block Table) → 5(GpuAllocator)

问：GpuAllocator (5) 依赖什么？
答：它需要 Device 知道在哪个设备上分配
    ↓
依赖：1(Device)
```

**练习**：对 Scheduler (35) 做依赖分析

```
答案：
Scheduler 依赖：
- Request Queue (39) - 提供待调度的请求
- KVCacheManager (19) - 了解缓存资源状态
- ModelRunner (13) - 知道模型能否接受新请求
```

### 方法二：数据流分析（Data Flow Analysis）

**核心问题**：数据是如何流过系统的？

**操作步骤**：

1. **找到输入和输出**
   - 输入：外部请求（API 调用）
   - 输出：生成的 token

2. **追踪数据变换**
   - 每个组件对数据做了什么变换？
   - 输入格式 → 变换 → 输出格式

3. **识别关键节点**
   - 哪些节点改变了数据的形态？
   - 哪些节点只是传递？

**vLLM 例子**：

```
输入：字符串 "Hello"
    ↓
Tokenize (23) → [1542, 345, ...]  ← 形态改变：字符串→数字
    ↓
Embedding (14) → [[0.1, 0.2, ...], ...]  ← 形态改变：ID→向量
    ↓
Transformer (15) → [[...], ...]  ← 形态改变：向量→向量
    ↓
PagedAttention (16) → 写入 KV Cache  ← 形态改变：计算结果→缓存
    ↓
Project (output layer) → [logits]  ← 形态改变：隐藏向量→概率
    ↓
Sampler (20) → 42  ← 形态改变：概率→Token ID
    ↓
Output：字符串 "World"
```

**关键洞察**：

- Tokenize 和 Embedding 是"编码"阶段
- Transformer 是"计算"阶段
- Sampler 是"决策"阶段
- 每个阶段的边界就是接口

### 方法三：最小系统分析（Minimum Viable System）

**核心问题**：要构建一个最简可运行系统，最少需要什么？

**操作步骤**：

1. **找到核心循环**
   - 系统最核心的业务是什么？
   - 对于推理系统：输入 → 计算 → 输出

2. **剥离所有"增强"功能**
   - 什么是必须有的？什么是可以没有但系统仍能运行的？
   - 比如：日志是必须的么？不是，删掉系统还能跑

3. **识别最小依赖链**
   - 沿着核心循环，逐个添加"没有这个就循环不起来"的组件

**vLLM 最小系统推导**：

```
Step 1: 核心循环
输入 → ? → 输出

Step 2: 需要计算
? = Model（模型必须存在）
输入 → Model → 输出

Step 3: Model 需要数据
Model 需要 Tensor
输入 → Model → Tensor → 输出

Step 4: Tensor 需要在哪运行？
Tensor 需要 Device（GPU）
输入 → Device → Model → Tensor → 输出

Step 5: 模型需要权重
Model 需要 Weights
输入 → Device → Model(Weights) → Tensor → 输出

Step 6: 需要采样输出
Tensor 需要 Sampler
输入 → Device → Model(Weights) → Tensor → Sampler → Token → 输出

Step 7: 需要循环
Token 需要循环回输入
输入 → ... → Token → (回到第一步)
```

**最小系统结果**（10个概念）：

```
Device (1)     - 硬件
Tensor (2)     - 数据
Model (12)     - 计算
Weights (29)   - 模型参数
Sampler (20)   - 决策
Token (23)     - 输入输出单元
+ 必要的辅助：KV Cache, GpuAllocator, Scheduler, API
```

### 方法四：接口契约分析（Interface Contract Analysis）

**核心问题**：模块之间如何协作？

**操作步骤**：

1. **识别每个模块的"边界"**
   - 什么进入模块？什么离开模块？
   - 这是模块的接口

2. **定义接口格式**
   - 输入的数据类型
   - 输出的数据类型
   - 可能的错误情况

3. **检查接口匹配**
   - 模块 A 的输出 = 模块 B 的输入？

**vLLM 例子**：

```
接口1：Tokenizer
输入：字符串 "Hello"
输出：List[int] [1542, 345, ...]

接口2：Embedding
输入：List[int] [1542, 345, ...]
输出：Tensor(shape=[3, hidden_size])

接口3：Transformer
输入：Tensor(shape=[batch, seq, hidden])
输出：Tensor(shape=[batch, seq, hidden])

接口4：Sampler
输入：Logits (Tensor, shape=[vocab_size])
输出：int (token_id)

接口匹配检查：
Tokenizer 输出 [1542, 345] → Embedding 需要 List[int] ✓
Embedding 输出 Tensor → Transformer 需要 Tensor ✓
Transformer 输出 Logits → Sampler 需要 Tensor ✓
```

### 方法五：分层架构分析（Layered Architecture Analysis）

**核心问题**：系统有哪些层次？每层负责什么？

**操作步骤**：

1. **按"负责对象"分组**
   - 谁为谁服务？
   - 基础设施层 → 业务逻辑层 → 服务层

2. **验证层内高内聚，层间低耦合**
   - 同层概念是否紧密相关？
   - 不同层概念是否相对独立？

**vLLM 分层**：

```
┌─────────────────────────────────────────────┐
│ 服务层 (40-45)                               │
│ 负责：对外提供服务                           │
│ vllm-engine | Engine API | OpenAI API       │
├─────────────────────────────────────────────┤
│ 调度层 (34-39)                               │
│ 负责：请求调度、批处理、排队                 │
│ Continuous Batching | Scheduler | Queue     │
├─────────────────────────────────────────────┤
│ 推理层 (20-29)                               │
│ 负责：模型执行、采样、优化                   │
│ Sampler | Forward | FlashAttention | ...    │
├─────────────────────────────────────────────┤
│ 模型层 (10-16)                               │
│ 负责：模型加载、计算                         │
│ Model | Embedding | Transformer | PagedAttn │
├─────────────────────────────────────────────┤
│ 资源层 (5,9,17-19,26)                        │
│ 负责：显存管理、缓存管理                     │
│ GpuAllocator | KVCache | BlockTable         │
├─────────────────────────────────────────────┤
│ 基础层 (0-4,6-8)                             │
│ 负责：最底层的原语                           │
│ Device | Tensor | Logger | Init | Config    │
└─────────────────────────────────────────────┘
```

## 二、分析流程实战

### 完整分析步骤

```
Step 1: 收集信息
├── 阅读官方文档
├── 查看源码结构
├── 搜索关键概念

Step 2: 识别核心业务
├── 核心功能是什么？
├── 输入是什么？输出是什么？

Step 3: 依赖分析
├── 从核心业务向外推
├── 哪些依赖哪些？

Step 4: 数据流分析
├── 数据如何流动？
├── 形态如何变化？

Step 5: 接口分析
├── 模块边界在哪里？
├── 接口匹配吗？

Step 6: 分层
├── 按职责分组
├── 验证层次清晰

Step 7: 验证
├── 最小系统是什么？
├── 能从 0 推演出来吗？
```

### 具体操作：用 vLLM 演示

**Step 1：收集信息**

```
vLLM 是什么？
- 大语言模型推理引擎
- 特点：PagedAttention、High Throughput
- 核心功能：接收文本输入，输出文本
```

**Step 2：识别核心业务**

```
核心业务：文本生成
输入：文本 prompt
输出：生成的文本

核心循环：
while not done:
    1. 预测下一个 token
    2. 将 token 加入输入
    3. 重复
```

**Step 3：依赖分析（手把手）**

```
问：预测下一个 token 需要什么？
答：需要知道之前的所有 token（KV Cache）

问：KV Cache 需要什么管理？
答：需要分配（Allocator）+ 调度（Manager）

问：计算需要什么？
答：需要模型（Model）+ 计算单元（Transformer）

问：模型从哪来？
答：需要加载（Loader）+ 注册（Registry）
```

**Step 4：数据流分析**

```
输入文本
    ↓ tokenize
Token IDs
    ↓ embedding
向量
    ↓ transformer layers
隐藏状态
    ↓ output layer
Logits（每个词的概率）
    ↓ sampler
Next Token
    ↓ 追加到输入
（循环）
    ↓ 结束
输出文本
```

**Step 5：接口分析**

```
每个箭头的两端就是一个接口：
- "tokenize" 接口：字符串 → [int]
- "embedding" 接口：[int] → [float]
- "transformer" 接口：[float] → [float]
- "sampler" 接口：[float] → int
```

**Step 6：分层**

```
按"谁为谁服务"分组：
- 底层：Device, Tensor, Allocator（为上层提供资源）
- 中层：Model, Transformer, Attention（执行计算）
- 上层：Scheduler, Batching（管理请求）
- 顶层：API（对外服务）
```

## 四、分析练习

### 练习 1：分析 Python 解释器

用今天学的方法，分析 Python 解释器。

```
核心功能：执行 Python 代码
输入：.py 文件
输出：程序行为

核心循环：
1. 解析代码（Parse）
2. 编译字节码（Compile）
3. 执行字节码（Execute）

依赖分析：
- Execute 依赖 Compile

分层：
- 词法/语法层

- 运行时层

```
核心功能：内存数据库

核心循环：
1. 接收命令

依赖分析：
- 执行命令依赖解析

## 五、常见陷阱

### 陷阱 1：陷入细节

**症状**：在分析早期就纠结于某个函数的实现细节
**解决**：先画粗粒度的架构，细节以后再补

### 陷阱 2：依赖环

**症状**：A 依赖 B，B 依赖 C，C 依赖 A
**解决**：找到环的入口点，打破循环，通常是有个"基础设施"不需要依赖其他

### 陷阱 3：分错层

**症状**：某个概念不知道放哪层
**解决**：问自己"它为谁服务？"答案就是它的上层

### 陷阱 4：过度工程

**症状**：想分析"完美"的依赖图
**解决**：够用就行，分析是为了理解，不是为了画图

## 六、进阶：自动推断

### 从源码推断

当你有源码时，如何快速分析？

1. **找入口点**：main 函数、run 函数
2. **追踪调用**：从入口向下走
3. **识别模式**：看到 "Manager", "Handler", "Factory" 这些词，通常是重要模块

```python
# 从类名推断职责
class *Manager:    # 资源管理
class *Handler:    # 处理请求
class *Factory:    # 创建对象
class *Cache:      # 缓存
class *Pool:       # 池化资源
class *Scheduler:  # 调度
class *Allocator:  # 分配
```

### 从报错推断

遇到错误时，如何快速定位？

```
错误信息 → 涉及哪些模块 → 这些模块依赖什么
```

例如：
```
CUDA OOM → KVCache 分配失败 → GpuAllocator → Device
```

## 七、总结：分析方法的层次

| 层次 | 方法 | 问题 | 输出 |
|------|------|------|------|
| 第一层 | 依赖分析 | X 依赖什么？ | 依赖图 |
| 第二层 | 数据流 | 数据怎么变？ | 数据流图 |
| 第三层 | 接口分析 | 模块怎么接？ | 接口契约 |
| 第四层 | 分层架构 | 谁服务谁？ | 层次结构 |
| 第五层 | 最小系统 | 最少需要什么？ | 核心模块集 |

这五个层次，**层层递进**，每上一层都基于下一层的分析结果。

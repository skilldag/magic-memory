## 设计理念

## 一、Debug 场景

Debug 场景训练你在遇到问题时，如何快速定位问题、分析原因、提出解决方案。

### 场景 1：CUDA OOM 错误

**场景描述**

你启动 vLLM 服务时一切正常，但当发送第二个请求后，服务崩溃并报错：

```
CUDA out of memory. Tried to allocate 2.00 GiB (GPU 0; 80.00 GiB total capacity; 
58.00 GiB already allocated; 1.50 GiB free; 59.50 GiB reserved in total by PyTorch)
```

**问题**

1. 这个错误涉及哪些核心概念？（至少列出 5 个编号）
2. 如果 KV Cache 配置过小，会导致什么问题？
3. 连续两个请求为什么会 OOM？应该增加还是减少 cache_size？
4. 除了调整 cache_size，还有哪些方法可以解决 OOM？

**答案要点**

1. 涉及 9(KV Cache)、19(KVCacheManager)、18(CacheBlock)、5(GpuAllocator)、0(VllmConfig)
2. KV Cache 配置过小会导致频繁的缓存驱逐，增加计算开销
3. 因为第一个请求占用了 KV Cache 空间，第二个请求没有足够空间。应该增加 cache_size 或启用缓存驱逐
4. 启用 Quantization (28)、减小 block_size、启用 Prefix Caching (38)、使用分布式部署 (50)

---

### 场景 2：推理速度很慢

**场景描述**

你的 vLLM 服务可以正常推理，但推理速度很慢。使用 `curl` 测试，发现每次推理需要 10 秒，而预期应该在 1 秒以内。

```
$ time curl -X POST  -d '{"prompt": "Hello", "max_tokens": 100}' 
> /dev/null
real    0m10.234s
```

**问题**

1. 推理慢可能涉及哪些概念？（从 0-50 中选择）
2. 如何判断是 Prefill 阶段慢还是 Decode 阶段慢？
3. 假设是 Decode 阶段慢，应该检查哪些配置？
4. 可以启用哪些特性来加速推理？

**答案要点**

1. 涉及 35(Scheduler)、34(Batching)、16(PagedAttention)、27(FlashAttention)、24(Decode Step)
2. 可以通过日志或 profiling 工具查看两个阶段的时间占比
3. 检查是否启用了 PagedAttention，检查 KV Cache 大小，检查是否启用了 FlashAttention
4. 启用 Continuous Batching、PagedAttention、FlashAttention、Speculative Decoding

---

### 场景 3：模型加载失败

**场景描述**

你尝试加载一个新的自定义模型，但 vLLM 报错：

```
ValueError: Model architecture "MyModel" is not supported. 
Please register it using `vllm.entrypoints.llm._register_model()`
```

**问题**

1. 这个错误涉及哪些核心概念？
2. 如何解决「模型未注册」的问题？
3. 如果模型加载后推理结果不对，应该检查什么？
4. 权重加载（29）失败有哪些可能原因？

**答案要点**

1. 涉及 10(ModelRegistry)、11(ModelLoader)、12(Model)
2. 需要在 ModelRegistry 中注册模型架构
3. 检查模型配置、权重是否正确加载、采样参数是否合理
4. 权重格式不支持、权重路径错误、磁盘空间不足

---

### 场景 4：请求排队时间长

**场景描述**

你发现某些请求需要等待很长时间才能开始处理。使用监控工具发现，Request Queue (39) 中有大量请求积压。

**问题**

1. 请求积压可能涉及哪些组件？（至少 4 个编号）
2. Scheduler (35) 的调度策略有哪些？
3. 如何调整 Continuous Batching (34) 来减少等待时间？
4. Prefill 优先 vs Decode 优先各适合什么场景？

**答案要点**

1. 涉及 39(Request Queue)、35(Scheduler)、34(Continuous Batching)、36(Prefill)、37(Decode)
2. 先来先服务、优先级队列、资源感知调度
3. 增加 max_num_seqs、调整 max_model_len
4. Prefill 优先适合长输入短输出，Decode 优先适合短输入长输出

## 二、优化场景

优化场景训练你如何根据业务需求和硬件条件，选择合适的优化策略。

### 场景 5：提升吞吐量

**场景描述**

你的服务需要处理大量并发请求，目前 QPS（每秒查询数）只有 10，需要提升到 100。

**问题**

1. 影响吞吐量的核心因素有哪些？
2. 应该启用哪些特性来提升吞吐量？（至少 5 个）
3. 如果请求都是短文本，应该调整什么配置？
4. 如果请求都是长文本，应该调整什么配置？

**答案要点**

1. 批处理效率、调度策略、缓存命中率、硬件利用率
2. 启用 Continuous Batching (34)、PagedAttention (16)、FlashAttention (27)、Prefix Caching (38)、Speculative Decoding (30)
3. 调整 max_model_len 较小值，增加 max_num_seqs
4. 增大 KV Cache，启用 Prefix Caching，调整 block_size

---

### 场景 6：降低显存占用

**场景描述**

你的 GPU 显存只有 16GB，但需要运行 70B 参数的模型。

**问题**

1. 显存主要用于哪些部分？（列出至少 4 个）
2. 哪些特性可以降低显存占用？
3. Quantization (28) 的精度选择有什么讲究？
4. 除了启用量化，还有什么方法？

**答案要点**

1. 模型权重、KV Cache、激活值、中间计算结果
2. 启用 Quantization、减小 KV Cache、启用梯度 checkpointing
3. INT8 精度损失小，INT4 精度损失大但节省更多显存
4. 使用模型并行、减少 batch size、减少 max_model_len

---

### 场景 7：降低延迟

**场景描述**

你的应用对延迟非常敏感，需要将首 token 时间从 2 秒降低到 0.5 秒。

**问题**

1. 首 token 时间主要受什么影响？
2. Prefill 阶段的优化重点是什么？
3. Speculative Decoding (30) 如何降低延迟？
4. 哪些量化方法对延迟影响较小？

**答案要点**

1. 模型加载时间、Prefill 阶段计算时间、调度等待时间
2. 减少输入长度、启用 FlashAttention、预热模型
3. 通过推测多个 token，减少自回归等待
4. INT8 AWQ、INT8 GPTQ 比 FP16 更快

---

### 场景 8：长上下文

**场景描述**

你需要处理 100K token 的长文档摘要任务，遇到了显存不足或性能下降的问题。

**问题**

1. 长上下文的主要挑战是什么？
2. 如何配置才能支持 100K token？
3. Prefix Caching (38) 在这个场景下有什么用？
4. 增大 KV Cache 的副作用是什么？

**答案要点**

1. KV Cache 显存占用巨大、计算复杂度高
2. 增大 cache_size、调整 max_model_len、使用更小的 block_size
3. 相同前缀（系统提示词）可以复用缓存，减少计算
4. 增加显存压力，可能导致 OOM

## 三、设计场景

设计场景训练你如何根据业务需求和技术约束，进行系统设计和技术选型。

### 场景 9：高并发服务设计

**场景描述**

你需要设计一个面向公众的 LLM API 服务，预计日均请求量 1000 万，峰值 QPS 5000。

**问题**

1. 需要考虑哪些高可用特性？（至少 5 个）
2. 应该使用哪些协议来暴露服务？（43、44、45）
3. 如何设计请求队列和调度策略？
4. 如何监控服务质量？

**答案要点**

1. 负载均衡、自动扩缩容、熔断限流、多实例部署
2. OpenAI API (43) 兼容外部调用，gRPC (44) 高效内部调用，WebSocket (45) 流式输出
3. 使用优先级队列、设置超时时间、启用 Continuous Batching
4. 监控 QPS、延迟、错误率、队列长度

---

### 场景 10：多模型部署

**场景描述**

你的产品需要同时提供 GPT、Llama、Mistral 三个模型供用户选择。

**问题**

1. 如何实现多模型支持？
2. 每个模型的资源如何隔离？
3. Multi-Lora (46) 在这里有用吗？
4. 调度器如何处理多模型请求？

**答案要点**

1. 使用 ModelRegistry (10) 注册多个模型，通过 Engine API (41) 切换
2. 使用容器或 cgroup 隔离资源，设置每个模型的显存配额
3. 如果需要微调版本，可以使用 Multi-Lora
4. 根据模型优先级和资源使用情况调度

---

### 场景 11：分布式推理

**场景描述**

你需要部署一个 70B 参数的模型，单卡无法加载。

**问题**

1. 分布式部署需要考虑哪些问题？
2. Tensor Parallel vs Pipeline Parallel 各适合什么场景？
3. 分布式下的 KV Cache 如何管理？
4. 如何处理节点间通信？

**答案要点**

1. 模型切分、通信开销、容错处理、负载均衡
2. Tensor Parallel 适合单节点多卡，Pipeline Parallel 适合多节点
3. 每节点维护本地 KV Cache，需要全局索引
4. 使用 NCCL 或 RDMA 加速通信

---

### 场景 12：安全与限流

**场景描述**

你的服务需要对用户进行限流，防止恶意请求和资源耗尽。

**问题**

1. 限流应该在哪个层级实现？（43 vs 35）
2. 可以使用哪些限流算法？
3. 如何防止某个请求占用过多 KV Cache？
4. 如何设置合理的超时时间？

**答案要点**

1. 可以在 API 层（43）或调度层（35）实现
2. 令牌桶、漏桶、固定窗口
3. 设置单个请求的最大 block 数量，启用驱逐策略
4. 根据模型大小和硬件性能设置，考虑 P95 延迟

## 四、综合场景

### 场景 13：完整方案设计

**场景描述**

你需要为一家 AI 创业公司设计 LLM 服务架构。公司有 3 台 8 卡 A100 服务器，需要服务 10 万用户，峰值同时在线 1000 人。

**问题**

1. 请设计完整的部署架构（硬件、资源分配）
2. 应该启用哪些优化特性？
3. 如何保证高可用和容错？
4. 如何监控和报警？

**答案要点**

1. 每台服务器部署多个 vLLM 实例，使用负载均衡
2. 启用 Continuous Batching、PagedAttention、FlashAttention、Quantization
3. 多副本部署、自动故障转移、资源隔离
4. 监控 QPS、延迟、GPU 利用率、显存使用、错误率

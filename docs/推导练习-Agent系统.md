# 推导练习：Agent 系统

> 用"从需求推导概念模型"的方法，手把手推导 Agent 系统

---

## Step 1: 明确需求

### 基础问题

```
Agent 系统要做什么？

输入：用户的自然语言指令（如"帮我订一张后天去北京的机票"）
输出：任务完成的结果（如"已订好机票，航班号XXX"）
```

### 核心特征

```
Agent vs 普通程序的区别：
- 自主性：能自己决定下一步做什么
- 主动性：不是等指令，而是主动规划
- 适应性：能根据反馈调整行动
- 工具使用：能调用外部工具完成任务
```

---

## Step 2: 抽象核心循环

### 核心问题：Agent 是如何工作的？

```
Agent 的本质是一个"思考-行动"循环：

while 任务未完成:
    1. 理解当前状态（Observation）
    2. 决定下一步做什么（Think/Plan）
    3. 执行行动（Action）
    4. 获取结果（Result）
    5. 判断是否完成

简化流程：
状态 → 思考 → 行动 → 结果 → (回到状态)
```

### 核心循环图

```
用户指令
    ↓
┌─────────────────────────────────────────────┐
│  while not done:                            │
│    1. 理解任务 (Understand)                 │
│    2. 制定计划 (Plan)                       │
│    3. 执行行动 (Execute)                    │
│    4. 观察结果 (Observe)                    │
│    5. 判断完成 (Check)                      │
└─────────────────────────────────────────────┘
    ↓
任务完成
```

## Step 3: 必要条件推导（关键！）

### 逐层追问

#### 第一层：要"制定计划"，需要什么？

```
问：Agent 是如何"制定计划"的？
答：需要有一个"大脑"来思考，这个大脑就是 LLM（大语言模型）

所以添加：LLM（推理引擎）
```

#### 第二层：LLM 需要什么输入？

```
问：LLM 需要什么才能制定计划？
答：需要知道：
    1. 用户要什么（用户指令）
    2. 之前做了什么（历史记忆）
    3. 能做什么（工具能力）

所以添加：
- Instruction Parser（指令解析器）- 理解用户意图
- Memory（记忆模块）- 存储历史
- Tools Registry（工具注册表）- 知道有哪些工具可用
```

#### 第三层：制定了计划后，如何执行？

```
问：LLM 制定了计划（比如"调用订票API"），谁来执行？
答：需要一个"执行器"来具体调用工具

所以添加：Action Executor（行动执行器）
    - 负责调用外部工具
    - 处理 API 请求
```

#### 第四层：执行后，如何知道结果？

```
问：执行完成后，Agent 如何知道做对了？
答：需要"观察"执行结果，把结果反馈给 LLM

所以添加：Observer（观察器）
    - 收集执行结果
    - 格式化后传给 LLM
```

#### 第五层：如何判断任务完成了？

```
问：Agent 怎么知道任务完成了？
答：需要"判断逻辑"，这个可以在 LLM 中做，也可以单独一个组件

这个可以内置在 LLM prompt 中，暂不单独列出
```

---

## 推导过程总结

```
需求：完成用户指令
    ↓
需要：制定计划（LLM）
    ↓
需要知道：用户指令 + 历史记忆 + 工具列表
    ↓
添加：Instruction Parser + Memory + Tools Registry
    ↓
需要：执行计划（Action Executor）
    ↓
需要：观察结果（Observer）
    ↓
需要：循环判断（内嵌在 LLM 中）
```

## Step 4: 补充辅助功能

### 除了核心，还需要什么？

```
1. 记忆管理
   - 短期记忆：当前对话的历史
   - 长期记忆：持久化的知识
   → 添加：Memory Manager

2. 工具管理
   - 注册工具
   - 工具描述（让 LLM 知道怎么用）
   → 添加：Tools Manager

3. 错误处理
   - 工具调用失败怎么办？
   - LLM 返回错误怎么办？
   → 添加：Error Handler

4. 配置管理
   - 用什么模型？
   - 最大循环次数？
   - 超时时间？
   → 添加：Config

5. 日志追踪
   - 记录执行过程
   - 方便调试
   → 添加：Logger
```

## Step 5: 组织层次结构

### 最终概念模型

```
┌─────────────────────────────────────────────────────────────────┐
│ 第六层：服务层（如何接收请求、返回结果）                         │
│ ┌─────────────┐ ┌─────────────┐                                │
│ │ API Handler │ │ Response    │                                │
│ │ (接收请求)   │ │ Formatter   │                                │
│ └─────────────┘ └─────────────┘                                │
├─────────────────────────────────────────────────────────────────┤
│ 第五层：协调层（管理整个 Agent 的运行）                         │
│ ┌─────────────┐ ┌─────────────┐                                │
│ │ Agent       │ │ Loop        │                                │
│ │ Controller  │ │ (循环控制)   │                                │
│ └─────────────┘ └─────────────┘                                │
├─────────────────────────────────────────────────────────────────┤
│ 第四层：推理层（理解、规划、决策）                              │
│ ┌─────────────┐ ┌─────────────┐ ┌─────────────┐               │
│ │ Instruction │ │ LLM         │ │ Tools       │               │
│ │ Parser      │ │ (大脑)      │ │ Selector    │               │
│ │ (理解指令)   │ │             │ │ (选择工具)   │               │
│ └─────────────┘ └─────────────┘ └─────────────┘               │
├─────────────────────────────────────────────────────────────────┤
│ 第三层：执行层（具体执行动作）                                  │
│ ┌─────────────┐ ┌─────────────┐                                │
│ │ Action      │ │ Tools       │                                │
│ │ Executor    │ │ Registry    │                                │
│ │ (执行器)     │ │ (工具注册)   │                                │
│ └─────────────┘ └─────────────┘                                │
├─────────────────────────────────────────────────────────────────┤
│ 第二层：观察层（获取反馈）                                      │
│ ┌─────────────┐ ┌─────────────┐                                │
│ │ Observer    │ │ Result      │                                │
│ │ (观察结果)   │ │ Parser      │                                │
│ └─────────────┘ └─────────────┘                                │
├─────────────────────────────────────────────────────────────────┤
│ 第一层：记忆层（存储信息）                                      │
│ ┌─────────────┐ ┌─────────────┐                                │
│ │ Memory      │ │ Knowledge   │                                │
│ │ (短期记忆)   │ │ Base        │                                │
│ └─────────────┘ └─────────────┘                                │
├─────────────────────────────────────────────────────────────────┤
│ 基础层（底层支持）                                              │
│ ┌─────────────┐ ┌─────────────┐ ┌─────────────┐               │
│ │ Config      │ │ Logger      │ │ Error       │               │
│ │ (配置)      │ │ (日志)      │ │ Handler     │               │
│ └─────────────┘ └─────────────┘ └─────────────┘               │
└─────────────────────────────────────────────────────────────────┘
```

## 最小 Agent 系统

如果要构建一个**最小可运行**的 Agent，只需要：

| 概念 | 作用 | 最少代码 |
|------|------|----------|
| LLM | 推理大脑 | 调用 API |
| Instruction Parser | 理解指令 | 正则解析 |
| Action Executor | 执行工具 | 函数调用 |
| Loop | 循环控制 | while 循环 |
| Observer | 获取反馈 | 返回值 |

**5 个概念**就能构建一个最小 Agent！

## 验证问题

### 问题 1：完整性

```
问：删掉某个模块，Agent 还能工作吗？

- 没有 LLM？→ 不能思考，不能规划
- 没有 Action Executor？→ 只能想，不能做
- 没有 Observer？→ 不知道做对了没有
- 没有 Memory？→ 每次都是新的对话
```

### 问题 2：接口匹配

```
Instruction Parser 输出 → LLM 输入 ✓
LLM 输出 → Action Executor 输入 ✓
Action Executor 输出 → Observer 输入 ✓
Observer 输出 → LLM 输入 ✓（循环）
```

### 问题 3：层次清晰

```
上层：LLM（思考）
中层：Executor（执行）
下层：Memory（存储）

思考依赖执行吗？不
执行依赖存储吗？不
但思考依赖存储（历史）
```

## 总结：Agent 推导结果

### 核心概念（必须）

| 编号 | 概念 | 作用 |
|------|------|------|
| 1 | LLM | 推理大脑 |
| 2 | Instruction Parser | 理解用户指令 |
| 3 | Tools Registry | 知道有哪些工具 |
| 4 | Action Executor | 执行具体动作 |
| 5 | Observer | 获取执行结果 |
| 6 | Loop | 循环控制 |

### 辅助概念（可选）

| 概念 | 作用 |
|------|------|
| Memory | 记忆历史 |
| Knowledge Base | 知识库 |
| Tools Manager | 工具管理 |
| Error Handler | 错误处理 |
| Config | 配置 |
| Logger | 日志 |

### 推导公式

```
用户需求 → 核心循环 → 必要条件 → 概念模型

核心循环：理解 → 规划 → 执行 → 观察 → 判断
必要条件：LLM + 工具 + 执行器 + 观察器
```

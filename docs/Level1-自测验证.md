## 答案提示（先思考再查看）

### 0 - VllmConfig

**答案 1**：model_config（模型配置）、cache_config（缓存配置）、scheduler_config（调度配置）
**答案 2**：影响 GpuAllocator、KVCacheManager、Block Table 大小
**答案 3**：Engine 创建之前，配置决定 Engine 的行为

### 1 - Device 抽象

**答案 1**：为了支持多种设备（CPU、GPU、不同厂商的加速卡），抽象解耦
**答案 2**：至少需要实现 device_type、allocate、deallocate 等核心方法
**答案 3**：Tensor 依赖 Device 来确定存储位置，Device 是 Tensor 的属性

### 2 - Tensor

**答案 1**：shape 定义维度，strides 定义每个维度跳过的字节数
**答案 2**：vLLM 的 Tensor 是轻量级数据结构，PyTorch 的 Tensor 是完整的张量对象
**答案 3**：可以是分散的，通过 strides 逻辑上连续

### 3 - Logger/Tracing

**答案 1**：高日志级别（如 DEBUG）会产生大量 IO，影响性能
**答案 2**：Tracing 有一定开销，关键路径会跳过
**答案 3**：日志看整体状态，tracing 看具体调用链

### 4 - vllm-core

**答案 1**：core 是基础依赖库，engine 和 entrypoint 依赖 core
**答案 2**：Device、Allocator、Tensor 等基础抽象是不可分割的

### 5 - GpuAllocator

**答案 1**：为了管理显存池、避免碎片化、处理分配失败
**答案 2**：碎片化处理、OOM 回退、显存配额控制
**答案 3**：尝试回退到 CPU，或者抛出 OOM 错误

### 6 - Error Handling

**答案 1**：VllmError 是领域特定错误，包含 vLLM 特定的错误码和上下文
**答案 2**：快速失败如 OOM 立即终止，优雅降级如部分请求失败继续处理其他请求
**答案 3**：通过 context manager 或 finally 块释放

### 7 - Init

**答案 1**：在所有组件初始化之前，不调用则使用默认配置
**答案 2**：重要，Logger 需要最先初始化，Allocator 需要在 Logger 之后

### 8 - Foundation Layer

**答案 1**：Foundation Layer 是所有其他层依赖的底层，不能被其他层替换
**答案 2**：Device、Allocator、Tensor 等基础抽象

### 9 - KV Cache

**答案 1**：Cache 意味着缓存复用，CPU Cache 是硬件级，KV Cache 是软件级
**答案 2**：受 GPU 显存大小、模型参数大小、运行时动态分配影响
**答案 3**：调度器会驱逐旧请求的缓存，或者等待

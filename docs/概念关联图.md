# 概念关联图

> vLLM 50 个概念的关联网络——从散点到网络

## 设计理念

知识不是孤立的点，而是相互连接的网络。本文档展示 50 个概念之间的两种关系：数据流关系（纵向）和依赖关系（横向）。理解这些关联，才能真正掌握 vLLM。

---

## 一、数据流关联（纵向）

数据流展示一个请求从进入 vLLM 到返回 token 经历的完整路径。这是理解 vLLM 工作原理的核心。

### 完整数据流路径

```
43(OpenAI API) → 41(Engine API) → 40(vllm-engine) → 35(Scheduler)
    → 34(Continuous Batching) → 13(ModelRunner) → 25(Forward Pass)
    → 14(Embedding) → 15(Transformer Layers) → 16(PagedAttention)
    → 9(KV Cache) → 17(Block Table) → 19(KVCacheManager)
    → 22(Logits) → 20(Sampler) → 21(Sampling Params)
    → 23(Token) → 24(Decode Step) → 返回新请求或结束
```

### 数据流分阶段解析

#### 阶段一：请求接入（入口）

```
43(OpenAI API) → 41(Engine API) → 40(vllm-engine)
```

这一阶段处理外部请求的接入。43 是对外的 HTTP 接口，41 是内部 API，40 是核心引擎。外部请求通过 43 进入，经过 41 解析和转换，最后交给 40 处理。

关键点：43 负责协议转换（OpenAI 协议 → vLLM 内部协议），41 负责请求验证和参数解析，40 负责协调各组件。

#### 阶段二：请求调度

```
40(vllm-engine) → 35(Scheduler) → 34(Continuous Batching) → 39(Request Queue)
```

这一阶段决定请求何时被处理。40 将请求交给 35 调度器，调度器根据策略决定处理顺序，34 负责将多个请求组织成批处理，39 负责排队。

关键点：35 是决策中心，34 是执行方式，39 是缓冲队列。

#### 阶段三：模型执行

```
13(ModelRunner) → 25(Forward Pass) → 14(Embedding) → 15(Transformer Layers) → 16(PagedAttention)
```

这一阶段是核心计算。13 是运行时封装，25 是一次前向计算，14 将 token 转为向量，15 是多层 Transformer，16 是注意力计算优化。

关键点：13 保护模型执行，25 是计算单元，16 是性能关键。

#### 阶段四：缓存管理

```
16(PagedAttention) → 9(KV Cache) → 17(Block Table) → 18(CacheBlock) → 19(KVCacheManager)
```

这一阶段管理注意力缓存。16 产生 KV 数据，9 存储 KV 数据，17 记录映射，18 是物理存储单位，19 是管理器。

关键点：16 是生产者，9 是存储结构，17 是索引，18 是物理载体，19 是调度。

#### 阶段五：采样输出

```
9(KV Cache) → 22(Logits) → 20(Sampler) → 21(Sampling Params) → 23(Token) → 24(Decode Step)
```

这一阶段决定输出什么。9 提供历史上下文，22 计算下一个 token 的概率，20 根据参数采样，23 是结果 token，24 是循环步骤。

关键点：22 是概率计算，20 是决策，24 是循环控制。

## 二、依赖关系（横向）

依赖关系展示哪些概念需要其他概念的支持。理解依赖关系，才能在出问题時快速定位根本原因。

### 核心依赖链

#### 依赖链一：配置 → 分配 → 存储 → 管理

```
0(VllmConfig) → 5(GpuAllocator) → 9(KV Cache) → 19(KVCacheManager)
```

关系解释：
- 0 定义配置参数（如 cache_size、block_size）
- 5 根据配置分配显存
- 9 是分配后的存储空间
- 19 管理和调度 9 的使用

如果出问题：先检查 0 的配置是否合理，再看 5 是否正确分配，然后看 9 是否足够，最后看 19 是否正确调度。

#### 依赖链二：注册 → 加载 → 模型 → 运行

```
10(ModelRegistry) → 11(ModelLoader) → 12(Model) → 13(ModelRunner)
```

关系解释：
- 10 注册模型元信息
- 11 加载模型权重
- 12 是模型计算本身
- 13 是执行环境

如果出问题：先检查 10 是否正确注册，再看 11 是否加载成功，然后看 12 是否正确，最后看 13 是否正常运行。

#### 依赖链三：设备 → 张量 → 分配 → 池

```
1(Device) → 2(Tensor) → 5(GpuAllocator) → 26(GPU Memory Pool)
```

关系解释：
- 1 抽象硬件设备
- 2 是数据表示
- 5 是分配器
- 26 是显存池

如果出问题：先检查 1 是否识别正确，再看 2 是否正确创建，然后看 5 是否正确分配，最后看 26 是否有足够空间。

#### 依赖链四：输入 → 嵌入 → 变换 → 注意力

```
23(Token) → 14(Embedding) → 15(Transformer) → 16(PagedAttention)
```

关系解释：
- 23 是原始输入
- 14 转换为向量
- 15 进行深度变换
- 16 计算注意力

如果出问题：先检查 23 是否正确分词，再看 14 是否正确嵌入，然后看 15 是否正确计算，最后看 16 是否正确优化。

#### 依赖链五：概率 → 采样 → 参数 → 结果

```
22(Logits) → 20(Sampler) → 21(Sampling Params) → 23(Token)
```

关系解释：
- 22 是原始概率
- 20 执行采样
- 21 控制采样行为
- 23 是最终结果

如果出问题：先检查 22 是否正确，再看 20 是否正常工作，然后看 21 参数是否合理，最后看 23 是否符合预期。

## 四、问题排查路径

当遇到特定问题时，应该按以下路径排查。

### OOM 问题排查

```
显存不足时，按以下顺序检查：

1. 检查 0(VllmConfig) 的 cache_size 配置
2. 检查 5(GpuAllocator) 的分配策略
3. 检查 9(KV Cache) 的大小
4. 检查 18(CacheBlock) 的大小和数量
5. 检查 19(KVCacheManager) 的驱逐策略
6. 检查 28(Quantization) 是否启用
7. 检查 50(Distributed) 是否需要启用
```

### 推理慢问题排查

```
推理延迟高时，按以下顺序检查：

1. 检查 35(Scheduler) 是否有积压
2. 检查 34(Continuous Batching) 的批处理策略
3. 检查 16(PagedAttention) 是否启用
4. 检查 27(FlashAttention) 是否启用
5. 检查 28(Quantization) 是否可以启用
6. 检查 30(Speculative Decoding) 是否可以启用
7. 检查 38(Prefix Caching) 是否可以利用
```

### 模型加载失败排查

```
模型加载失败时，按以下顺序检查：

1. 检查 10(ModelRegistry) 是否正确注册
2. 检查 11(ModelLoader) 的加载路径
3. 检查 29(Weights Loading) 的权重格式
4. 检查 47(GPU Driver) 是否正确安装
5. 检查 1(Device) 是否被正确识别
```

### 请求积压问题排查

```
请求排队时间长时，按以下顺序检查：

1. 检查 39(Request Queue) 的队列长度
2. 检查 35(Scheduler) 的调度策略
3. 检查 34(Continuous Batching) 的配置
4. 检查 36(Prefill) vs 37(Decode) 的资源分配
5. 检查 9(KV Cache) 是否足够
```

## 八、总结

### 核心观点

1. 数据流是纵向的，描述请求如何流动
2. 依赖关系是横向的，描述组件如何协作
3. 问题排查时，先查依赖链的上游
4. 性能优化时，组合多个相关特性

### 记忆技巧

- 记住三条主链路：接入→调度→执行→输出
- 记住四个核心依赖链
- 记住问题排查的检查顺序
